🧑 DATABASE BAGIAN USER
TABEL: users
Tabel utama pengguna sistem. Menyimpan semua akun dari berbagai role: Talent (kandidat promosi), Mentor (pembimbing), Atasan (supervisor langsung), Panelis (penguji independen), Finance, dan PDC Admin. Kolom mentor_id dan atasan_id adalah self-referencing FK yang menunjuk ke tabel users itu sendiri — artinya mentor dan atasan juga merupakan user dalam sistem. Saat login, sistem akan mengarahkan user ke halaman yang sesuai role-nya. Kolom role_id adalah role utama user, sedangkan tabel user_role menyimpan kemungkinan peran tambahan.
  • PK id bigint
  • -- username string, unique
  • -- password string
  • -- nama string
  • -- email string, unique
  • FK company_id bigint, nullable → company
  • FK department_id bigint, nullable → department
  • FK position_id bigint, nullable → position
  • FK role_id bigint → role
  • FK mentor_id bigint, nullable → users (self)
  • FK atasan_id bigint, nullable → users (self)
  • -- foto string, nullable
  • -- deleted_at timestamp (soft delete)
TABEL: user_role
Tabel pivot many-to-many antara users dan role. Digunakan karena satu orang bisa memiliki lebih dari satu peran. Contoh: seseorang yang menjabat sebagai Mentor sekaligus Atasan akan memiliki 2 baris di sini. Sistem menggunakan tabel ini untuk menentukan menu dan halaman apa saja yang boleh diakses user saat login. Kolom id_user → users, id_role → role.
  • PK id bigint
  • FK id_user bigint → users
  • FK id_role bigint → role
TABEL: company
Master data perusahaan/cabang. Satu perusahaan dapat memiliki banyak departemen (department) dan banyak karyawan (users). Kolom nama_company adalah nama resmi perusahaan. Mendukung multi-company sehingga sistem bisa dipakai oleh beberapa anak perusahaan sekaligus dalam satu platform.
  • PK id bigint
  • -- nama_company string
TABEL: department
Master data departemen di bawah sebuah perusahaan. Contoh: HRD, Finance, IT, Operations, dll. Kolom company_id (FK) mengikat departemen ke perusahaan induknya. Satu departemen hanya milik satu perusahaan, tetapi satu perusahaan bisa punya banyak departemen. User (karyawan) akan ditempatkan ke salah satu departemen via kolom department_id di tabel users.
  • PK id bigint
  • FK company_id bigint → company
  • -- nama_department string
TABEL: position
Master data jabatan/posisi dalam organisasi. Contoh: Staff, Supervisor, Officer, Manager, General Manager, BOD. Kolom grade_level adalah angka hierarki jabatan (1=Staff terendah, 5=BOD tertinggi) yang digunakan untuk menentukan arah promosi (selalu naik grade). Tabel ini direferensikan oleh: users (jabatan saat ini), development_sessions (jabatan asal & tujuan), promotion_plan (target jabatan), dan position_target_competence (standar kompetensi per jabatan).
  • PK id bigint
  • -- position_name string
  • -- grade_level tinyint 1=Staff, 5=BOD
TABEL: role
Master data peran/role pengguna. Berisi 6 role utama sistem: Talent (kandidat yang sedang diproses promosi), Mentor (pembimbing talent selama pengembangan), Atasan (supervisor langsung yang menilai talent), Panelis (penguji independen tahap akhir), Finance (memvalidasi aspek anggaran improvement project), dan PDC Admin (admin yang mengelola seluruh data sistem). Setiap role memiliki dashboard, menu, dan hak akses berbeda.
  • PK id bigint
  • -- role_name string
🚀 DATABASE BAGIAN DEVELOPMENT SESSION & PROMOSI
TABEL: development_sessions
⭐ Tabel HUB/PUSAT seluruh sistem IDP. Setiap baris mewakili satu siklus pengembangan talent dari jabatan asal ke jabatan tujuan. Semua tabel aktivitas (promotion_plan, assessment_session, idp_activity, improvement_project, panelis_assessments) memiliki FK ke tabel ini via kolom development_session_id. Ini memungkinkan semua aktivitas dalam satu periode dikelompokkan bersama. Satu talent bisa punya beberapa session jika siklus pengembangan diulang. Kolom source_position_id = jabatan awal, target_position_id = jabatan yang dituju. Kolom is_active menandai session yang sedang berjalan vs yang sudah selesai/diarsipkan.
  • PK id bigint
  • FK user_id_talent bigint → users
  • FK source_position_id bigint, nullable → position (jabatan asal)
  • FK target_position_id bigint, nullable → position (jabatan tujuan)
  • FK atasan_id bigint, nullable → users
  • -- mentor_ids json array ID mentor
  • -- status string In Progress, Promoted, dll
  • -- start_date / target_date date
  • -- is_active boolean
  • -- completed_at timestamp, nullable
  • -- deleted_at timestamp (soft delete)
TABEL: promotion_plan
Rencana promosi resmi talent. Satu baris = satu rencana promosi talent ke jabatan tertentu. Alur status lengkap: Draft → In Progress → Pending Panelis → Approved/Rejected Panelis → Ready → Promoted/Not Promoted. Jika sudah melalui sidang panelis, status diperinci menjadi: Ready Now, Ready in 1-2 Years, Ready in >2 Years, atau Not Ready. Kolom is_locked mencegah talent mengedit data setelah dikunci oleh admin. Kolom mentor_ids menyimpan array JSON ID mentor yang mendampingi (bisa lebih dari satu). Terhubung ke development_sessions sebagai konteks sesi pengembangan.
  • PK id bigint
  • FK user_id_talent bigint → users
  • FK target_position_id bigint → position
  • FK development_session_id bigint, nullable → development_sessions
  • -- mentor_ids json
  • -- status_promotion enum Draft, In Progress, Pending Panelis, Approved Panelis, Rejected Panelis, Ready, Promoted, Not Promoted, Ready Now, Ready in 1-2 Years, Ready in >2 Years, Not Ready
  • -- start_date / target_date date
  • -- is_locked boolean dikunci oleh admin
  • -- is_active boolean
TABEL: password_resets
Manajemen token reset password. Saat user meminta reset password, sistem membuat token unik (hash/MD5) dan menyimpannya di sini. Kolom expires_at membatasi masa berlaku token (misal: 1 jam). Kolom is_used = false artinya token belum dipakai; setelah token digunakan untuk ganti password, nilainya diubah ke true agar tidak bisa dipakai ulang (one-time use). Terhubung ke users via user_id, sehingga setiap permintaan reset diketahui milik siapa.
  • PK id bigint
  • FK user_id bigint → users
  • -- email string
  • -- token string varchar MD5/hash
  • -- expires_at timestamp batas berlaku token
  • -- is_used boolean false=belum, true=sudah dipakai
📋 DATABASE BAGIAN ASSESSMENT
TABEL: assessment_session
Sesi penilaian kompetensi antara talent dan atasan. Setiap baris merepresentasikan satu sesi assessment di mana atasan menilai talent secara komprehensif. Di dalam satu sesi, setiap kompetensi dinilai secara terpisah (lihat tabel detail_assessment). Kolom period menyatakan periode penilaian (misal: Q1-2026). Kolom user_id_atasan nullable karena pada beberapa kasus atasan belum ditentukan saat session dibuat. Kolom is_active memisahkan assessment aktif dari arsip session lama.
  • PK id bigint
  • FK user_id_talent bigint → users (yang dinilai)
  • FK user_id_atasan bigint, nullable → users (penilai)
  • FK development_session_id bigint, nullable → development_sessions
  • -- period string periode assessment
  • -- is_active boolean
TABEL: detail_assessment
Nilai per kompetensi dalam satu sesi assessment. Setiap baris menyimpan penilaian satu kompetensi dalam satu sesi. Kolom score_atasan: nilai yang diberikan atasan; score_talent: nilai self-assessment oleh talent sendiri; gap_score: selisih (score_atasan - score_talent), nilai positif = talent overestimate diri sendiri, nilai negatif = atasan tidak puas. notes: catatan penjelasan. Hasil gap score ini digunakan untuk menentukan area kompetensi mana yang perlu diprioritaskan dalam IDP.
  • PK id bigint
  • FK assessment_id bigint → assessment_session
  • FK competence_id bigint → competencies
  • -- score_atasan int nilai dari atasan
  • -- score_talent int self-assessment talent
  • -- gap_score decimal selisih nilai
  • -- notes string
TABEL: competencies
Master data daftar kompetensi yang diukur dalam sistem. Contoh kompetensi: Leadership, Communication, Problem Solving, Technical Skill, dll. Tabel ini menjadi titik referensi untuk 3 tabel lain: (1) detail_assessment — kompetensi apa yang dinilai dalam assessment, (2) question — soal kuesioner self-assessment per kompetensi, (3) position_target_competence — standar level minimum kompetensi yang dibutuhkan per jabatan.
  • PK id bigint
  • -- name string
TABEL: question
Bank soal kuesioner self-assessment berbasis kompetensi. Setiap soal terikat ke satu kompetensi dan memiliki tingkat level (1–5). Ketika talent mengisi self-assessment, sistem menampilkan pertanyaan-pertanyaan dari tabel ini berdasarkan level kompetensi yang sedang diukur. Jawaban talent menjadi dasar score_talent di tabel detail_assessment.
  • PK id bigint
  • FK competence_id bigint → competencies
  • -- level tinyint level soal (1-5)
  • -- question_text text, nullable
TABEL: position_target_competence
Standar kompetensi minimum per jabatan (pivot position ↔ competencies). Menentukan level kompetensi minimum yang harus dimiliki seseorang untuk menduduki jabatan tertentu. Contoh: posisi Manager memerlukan Leadership level 4, Communication level 3. Kolom target_level = nilai minimum yang harus dicapai. Data ini digunakan sistem untuk memperlihatkan "gap yang harus ditutup" oleh talent agar layak dipromosikan ke jabatan target.
  • PK id bigint
  • FK position_id bigint → position
  • FK competence_id bigint → competencies
  • -- target_level int level minimum yang harus dicapai
TABEL: panelis_assessments
Penilaian akhir oleh Panelis (penguji independen). Tahap ini terjadi setelah talent lolos dari assessment atasan. Panelis melakukan wawancara/sidang dan mengisi skor akhir di sini. Constraint UNIQUE pada (user_id_talent, panelis_id) memastikan satu panelis hanya menilai talent yang sama sekali dalam satu session. Kolom panelis_scores_json menyimpan rincian skor per aspek dalam format JSON. Kolom panelis_rekomendasi berisi keputusan akhir panelis apakah talent direkomendasikan promosi atau tidak.
  • PK id bigint
  • FK user_id_talent bigint → users (yang dinilai)
  • FK panelis_id bigint → users (penilai)
  • FK development_session_id bigint, nullable → development_sessions
  • -- panelis_score int, nullable
  • -- panelis_scores_json json detail skor per aspek
  • -- panelis_komentar text, nullable
  • -- panelis_rekomendasi text, nullable
  • -- panelis_tanggal_penilaian date, nullable
  • -- is_active boolean
📓 DATABASE BAGIAN IDP (Individual Development Plan)
TABEL: idp_type
Kategori/jenis metode pengembangan dalam IDP. Berisi 3 jenis utama: Exposure (paparan langsung ke situasi baru, misal: rotasi, magang, job shadowing), Mentoring (bimbingan tatap muka rutin dengan mentor), dan Learning (pelatihan formal seperti training, seminar, kursus). Setiap aktivitas logbook IDP wajib dikategorikan ke salah satu tipe ini.
  • PK id bigint
  • -- type_name string Exposure, Mentoring, Learning
TABEL: idp_activity
Logbook aktivitas pengembangan talent (inti dari IDP). Setiap baris = satu kegiatan yang dilakukan talent dalam rangka mengembangkan kompetensinya. Kolom penting: theme (tema kegiatan), activity_date (kapan dilakukan), location (tempat), activity (nama kegiatan), description (penjelasan detail), action_plan (rencana tindak lanjut), document_path & file_name (bukti berupa file). Setelah talent mengajukan, mentor/atasan memverifikasi: status = Pending → Approved/Rejected. Kolom verify_by mencatat siapa yang memverifikasi.
  • PK id bigint
  • FK user_id_talent bigint → users
  • FK type_idp bigint → idp_type
  • FK verify_by bigint, nullable → users (mentor/atasan)
  • FK development_session_id bigint, nullable → development_sessions
  • -- theme string tema kegiatan
  • -- activity_date date
  • -- location string
  • -- activity string
  • -- description text
  • -- action_plan text
  • -- document_path string
  • -- file_name string, nullable
  • -- status enum Pending, Approved, Rejected
  • -- platform string online/offline/hybrid
  • -- is_active boolean
🏗️ DATABASE BAGIAN IMPROVEMENT PROJECT
TABEL: improvement_project
Proyek perbaikan/inovasi nyata yang dikerjakan talent. Ini adalah "bukti kerja" konkret bahwa talent mampu menerapkan kompetensi yang dikembangkan. Talent mengajukan proposal proyek, lalu mentor/atasan memverifikasi, kemudian Finance meninjau aspek anggaran. Alur status: On Progress → Pending (diajukan) → Verified / Rejected. Kolom feedback = catatan dari mentor/atasan, finance_feedback = catatan dari Finance. ⚠️ Model juga menyimpan field panelis (score, komentar, rekomendasi, tanggal penilaian) sebagai data legacy yang mungkin akan dipindah ke tabel panelis_assessments.
  • PK id bigint
  • FK user_id_talent bigint → users (pemilik proyek)
  • FK verify_by bigint, nullable → users (mentor/atasan yang verifikasi)
  • FK development_session_id bigint, nullable → development_sessions
  • -- title string judul proyek
  • -- document_path string path file dokumen proyek
  • -- status enum On Progress, Pending, Verified, Rejected
  • -- is_active boolean session aktif/arsip
  • -- verify_at datetime, nullable waktu diverifikasi
  • -- feedback text, nullable catatan mentor/atasan
  • -- finance_feedback text, nullable catatan dari Finance
  • -- panelis_score int, nullable ⚠️ field legacy panelis
  • -- panelis_scores_json json, nullable ⚠️ rincian skor panelis
  • -- panelis_komentar text, nullable ⚠️ komentar panelis
  • -- panelis_rekomendasi text, nullable ⚠️ rekomendasi panelis
  • -- panelis_tanggal_penilaian date, nullable ⚠️ tanggal sidang panelis
🔧 DATABASE BAGIAN SISTEM & LOG
TABEL: status_log
Log riwayat perubahan status secara generic/global. Dirancang untuk merekam semua perubahan status di berbagai tabel (IDP Activity, Improvement Project, dll.) tanpa menggunakan FK langsung — nama tabel asal disimpan di kolom name_tabel sebagai teks. Kolom remark berisi alasan perubahan (misal: "Dokumen belum lengkap"). ⚠️ Catatan penting: Tabel ini saat ini belum aktif digunakan oleh kode backend — strukturnya sudah ada di database tapi belum ada controller/model yang menulis ke tabel ini. Fungsinya akan berguna jika diimplementasikan untuk menampilkan riwayat audit trail ke Atasan atau Mentor.
  • PK id bigint
  • -- name_tabel string nama tabel yang berubah
  • -- status enum Pending, Approved, Rejected
  • -- remark string, nullable alasan perubahan
TABEL: audit_activity
Log keamanan dan audit trail sistem. Merekam semua event penting untuk keperluan monitoring keamanan dan forensik. Jenis event yang direkam: login_success, login_failed, password_reset, role_change, file_upload, file_download, file_download_unauthorized, user_updated, dll. Kolom properties (JSON) menyimpan data konteks tambahan (nama file, perubahan data, dll.). Kolom ip_address dan user_agent membantu identifikasi apakah akses berasal dari perangkat/lokasi yang mencurigakan. Kolom user_id nullable karena event seperti login gagal terjadi sebelum user terautentikasi.
  • PK id bigint
  • FK user_id bigint, nullable → users (null jika belum login)
  • -- event string(100) login_success, file_upload, dll
  • -- description string, nullable
  • -- properties json, nullable data tambahan
  • -- ip_address string(45)
  • -- user_agent string
TABEL: app_notifications
Notifikasi dalam aplikasi untuk setiap user. Dikirim saat ada perubahan status IDP, approval, penolakan, dll. Bisa ditandai sudah dibaca (is_read).
  • PK id bigint
  • FK user_id bigint, nullable → users
  • -- title string
  • -- desc text
  • -- type string success, info, warning
  • -- is_read boolean
TABEL: documents
Penyimpanan metadata file yang diupload oleh user. Path file disimpan di sini (file fisik di storage). Digunakan untuk manajemen dokumen terpusat.
  • PK id bigint
  • FK user_id bigint → users
  • -- original_name string
  • -- file_path string, unique
  • -- mime_type string(150), nullable

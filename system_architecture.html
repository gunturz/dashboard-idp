4.1 Gambaran Umum Proyek

Diagram Arsitektur Backend - Dashboard IDP (Laravel 11 - PHP 8.2 - MySQL)

Gambar 3. Diagram Arsitektur Backend Sistem Dashboard IDP

Diagram ini sudah tepat untuk menggambarkan arsitektur backend proyek Dashboard IDP karena seluruh komponen yang ditampilkan memang berhubungan langsung dengan proses server-side yang ada di aplikasi ini. Alur utamanya dimulai dari browser milik pengguna dengan berbagai role, lalu request masuk ke aplikasi Laravel 11 melalui route, middleware, autentikasi, dan policy akses. Di titik ini backend bertugas memastikan setiap request hanya bisa diproses oleh user yang berhak, baik melalui middleware keamanan, verifikasi email, pembatasan role, maupun policy pada fitur-fitur sensitif seperti unduhan dokumen dan ekspor PDF. Dengan susunan seperti ini, diagram sudah mencerminkan bahwa inti sistem bukan berada di frontend, melainkan di lapisan backend yang mengatur logika akses, validasi, dan distribusi proses bisnis.

Bagian backend pada diagram juga sudah sesuai dengan karakter proyek ini karena controller dan Livewire diposisikan sebagai pusat pemrosesan logika bisnis. Ini sejalan dengan struktur kode yang memang banyak memakai controller per role dan komponen Livewire untuk antarmuka yang reaktif seperti tabel, daftar notifikasi, manajemen data, dan validasi workflow. Pada konteks proyek ini, Livewire bukan layanan terpisah, melainkan bagian dari aplikasi Laravel yang bekerja di sisi server untuk merespons interaksi pengguna tanpa harus berpindah ke arsitektur frontend SPA. Karena itu, penempatan Livewire di dalam blok backend sudah benar dan lebih representatif dibanding menaruhnya sebagai sistem independen.

Dari sisi data, diagram juga sudah cukup akurat karena MySQL menjadi basis data utama untuk menyimpan data operasional aplikasi, mulai dari user, role, sesi pengembangan, assessment, logbook, notifikasi, hingga data pendukung lain. Pemakaian driver database untuk session, cache, dan queue juga konsisten dengan konfigurasi proyek, sehingga penyebutan MySQL sebagai pusat penyimpanan backend tidak berlebihan. Sementara itu, file logbook dan project improvement memang lebih tepat disimpan di storage/app/private agar tidak bisa diakses langsung oleh browser. Alur aksesnya harus tetap melalui backend supaya file tersebut selalu melewati validasi otorisasi, policy, dan route aman sebelum diberikan ke pengguna yang berhak.

Layanan eksternal yang ditampilkan di diagram juga sudah relevan dengan isi proyek. Laravel Reverb memang digunakan untuk notifikasi real-time melalui event broadcast dengan channel privat per user, sehingga pengguna menerima pembaruan tanpa harus melakukan refresh manual. SMTP Gmail dipakai untuk email verifikasi dan reset password, Google OAuth 2.0 melalui Laravel Socialite digunakan untuk login Google, ClamAV memeriksa file yang diunggah sebelum disimpan, dan DomPDF merender laporan dari view Blade menjadi PDF. Dengan demikian, diagram ini secara struktur sudah benar untuk konteks backend, hanya saja jika ingin benar-benar sangat rapi untuk dokumen formal, Reverb bisa tetap dipertahankan sebagai layanan eksternal real-time, sementara penjelasan di bawah diagram perlu menekankan bahwa semua proses ini tetap dikendalikan oleh backend Laravel, bukan oleh frontend.
